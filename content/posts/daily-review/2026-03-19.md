## 오늘 다룬 기술

### 1. LLM Hallucination 방지 — 이원화 전략

LLM이 생성하는 응답에서 사실과 다른 내용(hallucination)이 포함되는 문제는 품질 평가 시스템에서 치명적이다. 이를 해결하기 위해 **판단 영역을 둘로 분리하는 이원화 전략**을 적용했다.

- **주관적 판단**: LLM이 담당한다. 문맥 이해, 뉘앙스 파악, 종합적 평가처럼 사람의 감각이 필요한 영역이다.
- **객관적 사실 검증**: 서버가 직접 데이터를 파싱하여 LLM의 판단을 오버라이드한다. 수치, 존재 여부, 규칙 기반 검증처럼 정답이 명확한 영역이다.

이 구조의 핵심은 LLM을 "만능 판단자"로 쓰지 않고, **잘하는 영역에만 집중시키는 것**이다.

### 2. 프롬프트 튜닝과 평가 기준 진화

4라운드에 걸친 프롬프트 튜닝을 통해 평가 기준을 v1.0에서 v1.5로 개선했다. 주요 변화는 **도메인 특화 표현을 범용 표현으로 전환**한 것이다. 특정 업종에서만 통하는 용어 대신 일반화된 기준을 사용함으로써, 하나의 프롬프트로 여러 업종에 적용 가능한 확장성을 확보했다.

### 3. 멀티서버 데이터 흐름 동기화

Java 서버(비즈니스 로직), Python 서버(분석/추론), React 프론트(사용자 인터페이스) 세 계층 간 데이터 흐름을 동기화했다. 서로 다른 언어와 역할을 가진 서버들이 일관된 데이터 상태를 유지하도록 조율하는 작업이다.

## 문제와 접근법

**문제 유형**: LLM 기반 시스템에서 "AI가 틀릴 수 있다"는 본질적 한계를 어떻게 제품 수준의 신뢰성으로 끌어올릴 것인가.

**접근법**: 완벽한 LLM을 만드는 대신, LLM이 틀릴 수 있는 영역을 구조적으로 차단했다. 객관적 사실은 LLM에게 맡기지 않고 서버가 직접 검증하여 덮어쓰는 방식이다. 이는 "AI를 신뢰하되 검증한다(Trust but Verify)"는 원칙의 구현이다.

프롬프트 측면에서는 반복적 실험(4라운드)을 통해 점진적으로 품질을 높이는 접근을 택했다. 한 번에 완벽한 프롬프트를 작성하려 하지 않고, 실제 결과를 보며 개선하는 실용적 방법론이다.

## 장단점 분석

| | 장점 | 단점 |
|---|------|------|
| **이원화 전략** | hallucination 리스크를 구조적으로 제거, LLM 비용 절감(불필요한 검증 요청 감소) | 어떤 항목이 "객관적"인지 사전 분류가 필요, 분류 기준이 모호한 경계 케이스 존재 |
| **프롬프트 범용화** | 멀티 도메인 확장이 쉬움, 프롬프트 관리 비용 감소 | 특정 도메인의 미묘한 차이를 놓칠 수 있음, 범용성과 정밀도 사이 트레이드오프 |
| **서버 오버라이드** | 확실한 정확도 보장, 감사(audit) 추적 용이 | 서버 파싱 로직 유지보수 부담, 새로운 검증 항목 추가 시 개발 필요 |

## 대안 접근법

- **RAG(Retrieval-Augmented Generation)**: LLM에게 검증용 근거 문서를 함께 제공하여 hallucination을 줄이는 방식. 이원화보다 구현이 단순할 수 있으나, 여전히 LLM이 근거를 무시할 가능성이 있다.
- **다중 LLM 교차 검증**: 서로 다른 모델로 같은 질문을 던져 결과를 비교하는 방식. 정확도는 높아지나 비용과 지연이 배로 증가한다.
- **구조화된 출력 강제(Structured Output)**: JSON Schema 등으로 LLM 출력 형식을 강제하여 파싱 실패를 줄이는 방식. 이원화 전략과 결합하면 시너지가 있다.
