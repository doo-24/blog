## 오늘 다룬 기술

### 1. 컨테이너 가상화 원리
Linux namespace(PID, NET, MNT, UTS, IPC, USER)를 통한 프로세스 격리와 cgroup을 이용한 리소스 제한이 컨테이너의 핵심이다. VM이 하드웨어를 가상화하는 것과 달리, 컨테이너는 호스트 커널을 공유하면서 프로세스 수준에서 격리한다. Union Filesystem(OverlayFS)의 레이어 구조가 이미지의 효율적인 저장과 배포를 가능하게 한다.

### 2. 멀티모달 LLM을 활용한 OCR
이미지를 base64로 인코딩하여 클라우드 기반 멀티모달 LLM에 전달하는 방식으로 텍스트를 추출하는 OCR 엔드포인트를 구현했다. 전통적인 OCR 엔진(Tesseract 등) 대신 LLM의 시각 이해 능력을 활용하는 접근법이다.

### 3. 빌드 파이프라인 분리
모놀리식 빌드 스크립트에서 프론트엔드 빌드 단계를 분리하여, 백엔드만 독립적으로 빌드할 수 있도록 변경했다. 개발 환경에서 불필요한 프론트엔드 빌드 대기 시간을 제거하기 위한 조치다.

## 문제와 접근법

**컨테이너 기술 문서화**: 컨테이너가 "가벼운 VM"이라는 오해를 풀기 위해, namespace → cgroup → Union FS → 런타임 아키텍처 순서로 계층적으로 설명하는 구조를 택했다. Dockerfile 최적화(멀티 스테이지 빌드, 레이어 캐시)와 Compose를 이용한 로컬 개발 환경 구성까지 실전 중심으로 정리했다.

**LLM 기반 OCR**: 기존 규칙 기반 OCR의 한계(레이아웃 인식 부족, 다국어 지원 미흡)를 멀티모달 LLM으로 해결했다. 이미지를 base64로 인코딩해 API 요청에 포함시키는 방식이다.

**빌드 디커플링**: Gradle 빌드 스크립트에서 프론트엔드 태스크 의존성을 제거하여, 백엔드 개발자가 Node.js 환경 없이도 서버를 실행할 수 있게 했다.

## 장단점 분석

| 접근법 | 장점 | 단점 |
|--------|------|------|
| LLM 기반 OCR | 레이아웃 이해력 우수, 다국어 자연스럽게 지원, 문맥 기반 추출 | API 호출 비용, 응답 지연, 대용량 이미지 처리 시 토큰 제한 |
| 빌드 파이프라인 분리 | 개발 사이클 단축, 환경 의존성 감소 | 배포 시 별도 통합 빌드 단계 필요, 프론트 변경이 반영 안 될 위험 |
| 컨테이너 계층적 설명 | 원리부터 실전까지 연결되는 학습 경로 | 분량이 많아 독자 이탈 가능 |

## 대안 접근법

- **OCR**: 전통적 OCR 엔진(Tesseract, PaddleOCR)과 LLM을 결합하는 하이브리드 방식 — OCR 엔진이 1차 추출하고 LLM이 후처리하면 비용을 절감하면서 정확도를 유지할 수 있다.
- **빌드 분리**: Gradle 조건부 태스크 대신 아예 별도 모듈(멀티 모듈 프로젝트)로 분리하면 더 깔끔한 의존성 관리가 가능하다.
- **문서화**: 단일 장문 포스트 대신 인터랙티브 다이어그램(Mermaid, D2)을 활용하면 시각적 이해도를 높일 수 있다.

## 다음에 공부할 것

1. **컨테이너 런타임 비교 (containerd vs CRI-O)** — Docker 너머의 컨테이너 생태계를 이해하기 위해. Kubernetes가 Docker를 런타임에서 제거한 이유와 연결된다.
2. **멀티모달 LLM의 비전 성능 벤치마크** — OCR 정확도, 비용, 지연 시간을 전통 OCR 엔진과 정량적으로 비교해볼 필요가 있다.
3. **Gradle 멀티 모듈 프로젝트 설계** — 모놀리식 빌드 스크립트의 근본적 해결책. 프론트/백/공통 모듈 분리 패턴을 학습하면 빌드 파이프라인 전체를 개선할 수 있다.
